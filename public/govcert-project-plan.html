Development Narrative
⚙️ Backend is Phase 2–3 Priority
Every "connected" integration in the frontend is currently a UI simulation. The backend translates those into real OAuth flows, database records, and data pipelines. This section documents every service, endpoint, and schema needed.
🔐
Authentication Service
User identity, sessions, multi-tenancy, role-based access control
Phase 2 · Week 1–2
Handles login for both Admin and Customer roles. Admins (advisors) can see all their clients. Customers can only see their own workspace. OAuth tokens are NEVER sent to the browser — they live server-side only.
POST/api/auth/loginEmail/password → JWT + refresh token
POST/api/auth/logoutInvalidate session, clear refresh token
POST/api/auth/refreshExchange refresh token for new JWT
POST/api/auth/sso/googleGoogle OAuth SSO flow
POST/api/auth/sso/microsoftMicrosoft OAuth SSO flow
GET/api/auth/meCurrent user + role + permissions
POST/api/auth/inviteAdmin sends customer invite link
users {
  id UUID PRIMARY KEY,
  email TEXT UNIQUE NOT NULL,
  role ENUM('admin','customer'),
  workspace_id UUID REFERENCES workspaces(id),
  hashed_password TEXT, -- bcrypt, null if SSO-only
  refresh_token_hash TEXT, -- stored hashed
  last_login TIMESTAMP
}
🔗
OAuth Proxy Service
Handles all third-party OAuth token exchange — tokens NEVER reach the browser
Phase 2 · Week 2–4
The browser never sees access tokens. The frontend redirects to the backend's /oauth/start, which redirects to the provider, receives the code, exchanges it for tokens server-side, encrypts and stores them, then redirects back to the frontend with a session cookie only.
OAuth Endpoints
GET/api/oauth/start/:providerRedirect to provider auth
GET/api/oauth/callback/:providerReceive code, exchange for token
POST/api/oauth/refresh/:providerServer-side token refresh
DEL/api/oauth/revoke/:providerDisconnect + delete stored tokens
GET/api/oauth/statusWhich providers are connected
Supported Providers

📒 FreshBooksOAuth 2.0
🔵 XeroOAuth 2.0
🏦 PlaidPlaid Link

oauth_tokens {
  id UUID PRIMARY KEY,
  client_id UUID REFERENCES clients(id),
  provider TEXT, -- 'quickbooks' | 'gusto' | etc
  access_token_enc TEXT, -- AES-256 encrypted
  refresh_token_enc TEXT, -- AES-256 encrypted
  expires_at TIMESTAMP,
  provider_entity_id TEXT, -- e.g. QuickBooks realm ID
  scopes TEXT[] -- granted scopes
}

Data Sync Service
Pulls, normalizes, and caches data from all connected providers into a unified schema
Phase 2–3
Each provider returns data in a different format. This service translates all of them into GovCert's internal schema, so the frontend always works with consistent data regardless of the source. Data is cached in Redis to avoid repeated API calls.

GET/api/data/revenue3yr revenue normalized
GET/api/data/balance-sheetAssets, liabilities, net worth
GET/api/data/financials/verifyCross-check QB vs IRS data
Payroll Sync
POST/api/sync/payrollTrigger Gusto/ADP/Paychex pull
GET/api/data/employeesEmployee roster + classifications
GET/api/data/payroll-summaryAnnual payroll totals
Gov Data Sync
POST/api/sync/samPull SAM.gov entity by UEI
POST/api/sync/fpdsPull contract history from FPDS

financial_snapshots { client_id, fiscal_year, revenue, net_income, total_assets, liabilities, net_worth, source, synced_at }
employee_records { client_id, count_ft, count_pt, count_1099, annual_payroll, source, synced_at }

AI Service (Claude API)
Server-side Claude API calls — never expose the API key to the browser
Phase 3
All Claude API calls go through the backend. The frontend sends a "draft narrative" request with the client_id; the backend fetches the client's data, constructs the prompt, calls Claude, and returns the draft. API key never leaves the server.
POST/api/ai/draft/business-descriptionDraft 8(a) Section II-A narrative
POST/api/ai/draft/social-disadvantageDraft social disadvantage narrative framework
POST/api/ai/draft/quality-control-planDraft GSA MAS QCP section
POST/api/ai/draft/relevant-experienceDraft past performance narrative
POST/api/ai/score-applicationScore application 0–100 with suggestions
POST/api/ai/chatIn-app advisor chat (streaming)
Prompt architecture: Each AI endpoint assembles a structured prompt from the client's live data — revenue figures from the financial snapshot, employee data from the payroll sync, entity data from SAM.gov. This ensures narratives are always factually grounded in verified data, not hallucinated.

Technology Decisions
Recommended stack for GovCert backend. These are starting points — all choices are open for discussion.
LayerRecommendedWhyAlternative
Backend LanguageNode.js (TypeScript)Same ecosystem as frontend; huge library support for OAuth + APIs; fast I/O for data syncPython/FastAPI (good for AI-heavy), Go (better performance)
API FrameworkExpress.jsMost familiar, massive ecosystem, plenty of OAuth middlewareFastify (faster), Hono (edge-compatible)
Primary DBPostgreSQLRelational data fits perfectly; row-level security for multi-tenancy; great JSON supportMySQL, SQLite (simpler but less powerful)
ORMPrismaTypeScript-native, auto-generates types, easy migrationsDrizzle (lighter), Sequelize (older)
Cache / SessionsRedisSession storage, API response caching, token expiry TTLsMemcached, in-memory (not production-safe)
Hosting (Backend)Railway or RenderSimple Node.js deployment, managed Postgres + Redis, free tier, no DevOps overheadAWS (more powerful, more complex), Vercel (serverless, limitations with long-running jobs)
Hosting (Frontend)Netlify or VercelStatic hosting, CDN, easy custom domainsAWS S3 + CloudFront, Railway
Document StorageCloudflare R2S3-compatible, no egress fees, cheaper than AWS S3AWS S3, Supabase Storage
EmailSendGridReliable deliverability, templates, tracking, free tierResend (simpler API), Postmark
Auth StrategyJWT + Refresh TokensStateless, scalable; refresh tokens allow long sessions without sacrificing securitySessions only (simpler, less scalable)
BillingStripeIndustry standard, handles subscriptions, invoicing, metered billingPayPal, manual invoicing
